Этап 1. Доменная модель и состояния

Назначение документа



Документ фиксирует минимальную доменную модель этапа 1 и набор состояний, которые нужны для реализации первого сквозного сценария: регистрация пользователя, создание рабочего аккаунта, подключение Ozon, проверка соединения и запуск первичной инициализации.



Цель документа — до начала реализации убрать неопределённость в том, какие сущности существуют в системе, как они связаны между собой и какие состояния допустимы на этапе 1.



Общий принцип моделирования



На этапе 1 модель должна быть достаточно строгой, чтобы backend и frontend одинаково понимали состояние системы, но при этом не перегруженной будущими сценариями, которые относятся к следующим этапам.



На этом этапе доменная модель строится вокруг пяти сущностей:



User

Session

SellerAccount

OzonConnection

SyncJob



Этого достаточно, чтобы реализовать первый onboarding-flow без преждевременного усложнения.



Общая логика связей между сущностями



На этапе 1 принимается следующая связь сущностей:



один User владеет одним SellerAccount;

один SellerAccount имеет одно OzonConnection;

один SellerAccount может иметь несколько SyncJob, но на этапе 1 фактически используется только сценарий initial\_sync;

один User может иметь несколько Session.



Это даёт понятную и устойчивую схему:



User → SellerAccount → OzonConnection → SyncJob



1\. Сущность User

Назначение



User — это субъект аутентификации в системе.

Через эту сущность реализуются регистрация, логин, проверка текущего пользователя и управление доступом к защищённой части продукта.



User не является ни кабинетом Ozon, ни seller account, ни бизнес-объектом маркетплейса. Это именно учётная запись человека или клиента в нашей системе.



Минимальные поля

id

email

password\_hash

status

created\_at

updated\_at

Семантика полей

id



Внутренний первичный идентификатор пользователя.



email



Уникальный email пользователя, используемый для входа.



password\_hash



Хэш пароля. Пароль в открытом виде не хранится.



status



Состояние пользователя в системе.



created\_at



Дата и время создания пользователя.



updated\_at



Дата и время последнего обновления записи.



Бизнес-правила

email должен быть уникальным;

email должен храниться в нормализованном виде;

пароль хранится только в виде hash;

пользователь со статусом disabled не должен иметь возможности войти в систему.

Статусы User

active



Пользователь активен и может входить в систему.



disabled



Пользователь отключён и не должен получать доступ.



Что не входит в User на этапе 1



На этапе 1 в User не добавляются:



профильные поля;

имя/фамилия;

ролевой доступ;

настройки уведомлений;

organization/team membership.

2\. Сущность Session

Назначение



Session нужна для управления авторизацией пользователя после логина.

Даже если позже будет использован JWT-подход, на этапе 1 рекомендуется иметь отдельную session model, чтобы:



понимать, какие сессии активны;

уметь делать logout;

уметь истекать сессии;

не хранить “магическое состояние” только на стороне клиента.

Минимальные поля

id

user\_id

token\_hash или session\_id

expires\_at

created\_at

revoked\_at

Семантика полей

id



Внутренний идентификатор сессии.



user\_id



Ссылка на пользователя, которому принадлежит сессия.



token\_hash / session\_id



Идентификатор сессии. Если используется cookie-based auth, в БД должен храниться не сам сырой токен, а его безопасный идентификатор или hash.



expires\_at



Дата и время истечения сессии.



created\_at



Дата и время создания сессии.



revoked\_at



Дата и время отзыва сессии. Если поле не заполнено, сессия считается неотозванной.



Бизнес-правила

сессия принадлежит одному пользователю;

сессия считается валидной, если:

не истекла;

не отозвана;

принадлежит активному пользователю;

logout должен инвалидировать текущую сессию.

Что не входит в Session на этапе 1

device fingerprinting;

IP-аналитика;

suspicious login detection;

управление всеми сессиями пользователя из UI.

3\. Сущность SellerAccount

Назначение



SellerAccount — это рабочий бизнес-аккаунт клиента внутри нашей системы.

Это не эквивалент Ozon account, а именно наш внутренний объект, к которому привязываются интеграции, статусы onboarding и будущие данные.



Очень важно не смешивать User и SellerAccount.



User — сущность доступа и авторизации.

SellerAccount — сущность бизнес-контекста.

Минимальные поля

id

user\_id

name

status

created\_at

updated\_at

Семантика полей

id



Внутренний идентификатор seller account.



user\_id



Ссылка на владельца seller account.



name



Человеко-читаемое название аккаунта. На этапе 1 может генерироваться автоматически или быть задано пользователем позже.



status



Состояние рабочего аккаунта.



created\_at



Дата создания seller account.



updated\_at



Дата последнего обновления.



Бизнес-правила

на этапе 1 один пользователь имеет один seller account;

seller account может создаваться автоматически при регистрации;

seller account является контейнером для Ozon connection и sync jobs.

Статусы SellerAccount

onboarding



Аккаунт создан, но настройка ещё не завершена.



active



Аккаунт полностью активен и готов к работе.



archived



Аккаунт архивирован и не участвует в активной работе.



Рекомендуемая логика статусов

после регистрации seller account создаётся в статусе onboarding;

после успешного подключения и первичной инициализации seller account может стать active.

4\. Сущность OzonConnection

Назначение



OzonConnection описывает техническое подключение seller account к Ozon Seller API.



Эта сущность не хранит бизнес-данные Ozon. Она хранит:



credentials;

статус их проверки;

результат последней проверки;

состояние первичной инициализации на уровне подключения.

Минимальные поля

id

seller\_account\_id

client\_id\_encrypted

api\_key\_encrypted

status

last\_check\_at

last\_check\_result

last\_error

created\_at

updated\_at

Семантика полей

id



Идентификатор подключения.



seller\_account\_id



Ссылка на seller account.



client\_id\_encrypted



Зашифрованное значение Client ID.



api\_key\_encrypted



Зашифрованное значение API key.



status



Текущее состояние подключения.



last\_check\_at



Дата и время последней проверки подключения.



last\_check\_result



Краткий результат последней проверки. Может использоваться для отображения в UI.



last\_error



Последняя техническая ошибка или причина невалидности подключения.



created\_at



Дата создания подключения.



updated\_at



Дата обновления подключения.



Бизнес-правила

на этапе 1 один seller account имеет одно активное OzonConnection;

Client ID и API key должны храниться только в зашифрованном виде;

API key не должен возвращаться во frontend после сохранения;

изменение credentials должно сбрасывать валидность подключения до повторной проверки.

Статусы OzonConnection



Ниже — рекомендуемый минимальный набор состояний.



draft



Подключение создано, credentials сохранены, но проверка ещё не выполнена.



checking



В данный момент выполняется проверка credentials.



valid



Последняя проверка подключения завершилась успешно.



invalid



Последняя проверка подключения завершилась неуспешно.



sync\_pending



Подключение валидно, initial sync создан и ожидает выполнения.



sync\_in\_progress



Подключение валидно, initial sync выполняется.



sync\_failed



Во время initial sync произошла ошибка.



Как трактовать состояния подключения

Что считается “подключение создано”



Подключение считается созданным, если:



в БД есть запись OzonConnection;

credentials сохранены;

статус находится как минимум в draft.

Что считается “подключение валидно”



Подключение считается валидным, если:



была выполнена проверка credentials;

проверка завершилась успешно;

статус подключения установлен в valid.

Что считается “подключение невалидно”



Подключение считается невалидным, если:



проверка выполнялась;

credentials не прошли проверку;

статус установлен в invalid.

Что происходит при обновлении credentials



Если пользователь меняет Client ID или API key:



старый status должен сбрасываться;

подключение снова переходит в draft;

last\_check\_result и last\_error должны быть обновлены в соответствии с новой проверкой.

5\. Сущность SyncJob

Назначение



SyncJob фиксирует асинхронную работу системы по первичной инициализации.



На этапе 1 эта сущность нужна не для полной модели ingestion, а для того, чтобы:



можно было запустить initial\_sync;

можно было хранить состояние выполнения;

frontend мог показывать пользователю статус.

Минимальные поля

id

seller\_account\_id

type

status

started\_at

finished\_at

error\_message

created\_at

Семантика полей

id



Идентификатор sync job.



seller\_account\_id



Ссылка на seller account.



type



Тип задачи. На этапе 1 допустимое значение — initial\_sync.



status



Текущее состояние выполнения задачи.



started\_at



Дата и время фактического старта выполнения.



finished\_at



Дата и время завершения выполнения.



error\_message



Ошибка, если выполнение завершилось неуспешно.



created\_at



Дата и время постановки задачи.



Статусы SyncJob

pending



Задача создана и ожидает выполнения.



running



Задача выполняется.



completed



Задача завершилась успешно.



failed



Задача завершилась с ошибкой.



Связь между OzonConnection и SyncJob



На уровне продукта пользователь ожидает, что “запуск initial sync” относится к подключению Ozon.

Но на уровне доменной модели лучше оставить SyncJob отдельной сущностью, а OzonConnection использовать как источник текущего интеграционного состояния.



Практически это означает:



OzonConnection.status показывает высокоуровневое состояние подключения;

SyncJob.status показывает состояние конкретного запуска initial sync.

Рекомендуемое согласование статусов

Когда initial sync создаётся

создаётся SyncJob(status = pending)

OzonConnection.status = sync\_pending

Когда initial sync стартует

SyncJob.status = running

OzonConnection.status = sync\_in\_progress

Когда initial sync завершается успешно

SyncJob.status = completed

OzonConnection.status может вернуться в valid

SellerAccount.status может стать active

Когда initial sync падает

SyncJob.status = failed

OzonConnection.status = sync\_failed

ER-логика модели в текстовом виде



Ниже — текстовая схема связей.



User



Один пользователь может иметь много сессий.

Один пользователь на этапе 1 имеет один seller account.



Session



Каждая сессия принадлежит одному пользователю.



SellerAccount



Каждый seller account принадлежит одному пользователю.

Каждый seller account на этапе 1 имеет одно Ozon connection.

Каждый seller account может иметь несколько sync jobs.



OzonConnection



Каждое подключение принадлежит одному seller account.



SyncJob



Каждая sync job принадлежит одному seller account.

На этапе 1 используется только тип initial\_sync.



Текстовая ER-схема

User (1) ---- (N) Session

User (1) ---- (1) SellerAccount

SellerAccount (1) ---- (1) OzonConnection

SellerAccount (1) ---- (N) SyncJob

Минимальные переходы состояний

User

active -> disabled

SellerAccount

onboarding -> active

onboarding -> archived

active -> archived

OzonConnection

draft -> checking

checking -> valid

checking -> invalid

valid -> sync\_pending

sync\_pending -> sync\_in\_progress

sync\_in\_progress -> valid

sync\_in\_progress -> sync\_failed

invalid -> draft   (после обновления credentials)

valid -> draft     (после обновления credentials)

SyncJob

pending -> running

running -> completed

running -> failed

Что должен понимать frontend из этой модели



Frontend на этапе 1 не должен догадываться о состоянии системы “по косвенным признакам”.

Он должен получать явные состояния и строить UI на их основе.



Из User



Frontend понимает:



пользователь авторизован или нет.

Из SellerAccount



Frontend понимает:



onboarding завершён или ещё идёт.

Из OzonConnection



Frontend понимает:



подключение ещё не создано;

credentials сохранены, но не проверены;

подключение валидно;

подключение невалидно;

идёт initial sync;

initial sync завершился ошибкой.

Из SyncJob



Frontend понимает:



конкретный запуск initial sync ожидает выполнения;

выполняется;

завершён;

завершён с ошибкой.

Что важно не усложнять на этапе 1



Чтобы не перегрузить модель раньше времени, на этапе 1 не нужно вводить:



несколько seller account на одного пользователя;

несколько Ozon connections на один seller account;

отдельную сущность marketplace;

сложную модель ownership/team access;

отдельные сущности для credentials history;

сложную модель sync pipeline с множеством типов job.



Эти вещи могут понадобиться позже, но сейчас они только усложнят реализацию.



Итоговая минимальная модель этапа 1



На этапе 1 доменная модель считается достаточной, если в системе есть:



пользователь как субъект auth;

сессия как субъект авторизации;

seller account как субъект бизнес-контекста;

Ozon connection как субъект подключения внешнего кабинета;

sync job как субъект асинхронной первичной инициализации.



Именно эта модель обеспечивает первый сквозной сценарий без преждевременного усложнения.
