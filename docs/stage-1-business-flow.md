Этап 1. Business flow и ограничения

Цель этапа



Этап 1 нужен для реализации первого сквозного пользовательского сценария: пользователь заходит в систему, создаёт свою учётную запись, подключает кабинет Ozon и доводит систему до состояния, в котором подключение подтверждено и подготовлено к первичной инициализации.



Результат этапа должен быть не в виде разрозненных API и экранов, а в виде единого работающего onboarding-flow, который пользователь может пройти самостоятельно без ручных действий со стороны команды разработки.



Бизнес-цель этапа



На данном этапе продукт должен впервые начать выполнять свою базовую прикладную функцию: принимать пользователя, связывать его с его рабочим аккаунтом в системе и давать возможность безопасно подключить внешний кабинет Ozon.



Этап 1 не решает задачи аналитики, рекомендаций или загрузки бизнес-данных. Его задача — создать управляемую и безопасную точку входа в продукт, после которой можно переходить к интеграционным и аналитическим сценариям.



Scope этапа



В этап 1 входит реализация сценария регистрации, входа, создания пользовательского аккаунта в системе, сохранения подключения Ozon, проверки валидности credentials и запуска первичной инициализации.



Этап считается завершённым, если пользователь может:



зарегистрироваться;

войти в систему;

получить свой seller account;

добавить Client ID и API key Ozon;

сохранить подключение;

проверить, что подключение действительно работает;

увидеть статус подключения;

запустить первичную инициализацию;

увидеть статус первичной инициализации.

Happy path пользовательского сценария



Ниже описан основной целевой сценарий, который должен поддерживаться после завершения этапа.



Шаг 1. Регистрация



Пользователь открывает продукт и создаёт учётную запись, используя email и пароль.



Шаг 2. Логин



После регистрации пользователь входит в систему и получает авторизованную сессию.



Шаг 3. Создание или получение seller account



После успешной регистрации пользователь получает свой seller account в системе. В MVP seller account может создаваться автоматически и не требовать отдельного явного действия со стороны пользователя.



Шаг 4. Открытие экрана подключения Ozon



Пользователь переходит на экран подключения Ozon и видит форму для ввода Client ID и API key.



Шаг 5. Сохранение credentials



Пользователь вводит Client ID и API key, после чего система сохраняет их безопасным способом.



Шаг 6. Проверка подключения



Пользователь запускает проверку подключения. Система выполняет техническую валидацию credentials через Ozon API.



Шаг 7. Отображение статуса



Система показывает результат проверки подключения. Пользователь должен видеть понятный статус: подключение валидно или невалидно.



Шаг 8. Запуск первичной инициализации



Если подключение успешно подтверждено, пользователь может запустить первичную инициализацию.



Шаг 9. Отображение статуса инициализации



Система показывает, что первичная инициализация запущена, выполняется, завершилась успешно или завершилась с ошибкой.



Что считается результатом этапа



Этап 1 считается успешным, если пользователь проходит весь сценарий от регистрации до подтверждённого подключения Ozon без ручных действий через базу данных, Postman или административные инструменты.



Ключевой результат этапа — не “наличие auth” и не “наличие формы подключения”, а именно рабочий сквозной flow:



регистрация → вход → seller account → Ozon credentials → проверка подключения → запуск initial sync



Что входит в backend scope



На backend в рамках этапа 1 должны быть реализованы:



регистрация пользователя;

логин пользователя;

управление пользовательской сессией;

получение текущего пользователя;

сущность пользователя;

сущность seller account;

сущность подключения Ozon;

безопасное хранение Client ID и API key;

API сохранения подключения Ozon;

API проверки подключения;

API запуска первичной инициализации;

API получения текущего статуса подключения и initial sync.

Что входит во frontend scope



На frontend в рамках этапа 1 должны быть реализованы:



экран регистрации;

экран логина;

защищённая часть приложения после входа;

экран подключения Ozon;

форма ввода Client ID и API key;

отображение статуса подключения;

экран или блок первичной инициализации;

отображение статуса initial sync.

Что не входит в этап 1



Чтобы этап не расползался, нужно явно зафиксировать out of scope.



В этап 1 не входят:



загрузка всех бизнес-данных Ozon;

построение каталога товаров;

загрузка заказов, продаж, остатков и рекламных данных как завершённая функция;

аналитика и дашборды по Ozon;

складская логика;

рекламная логика;

ценообразование;

AI-чат;

рекомендации;

multi-marketplace;

поддержка нескольких seller account на одного пользователя;

поддержка нескольких Ozon-подключений внутри одного seller account;

продвинутый onboarding с несколькими сценариями;

ролевой доступ внутри клиентского аккаунта;

полноценная админка управления пользователями.

Ограничения MVP этапа 1



Для MVP в рамках этапа 1 принимаются следующие ограничения:



Один пользователь — один seller account



Каждый пользователь работает только со своим одним seller account. Сценарии групповой работы, нескольких рабочих пространств или выбора аккаунта не реализуются.



Один seller account — одно Ozon connection



В рамках MVP seller account может иметь только одно активное подключение Ozon.



Только Ozon Seller API



На этапе 1 поддерживается только подключение Ozon. Другие маркетплейсы и внешние системы не рассматриваются.



Проверка подключения — это не синхронизация



Проверка подключения означает только техническую валидацию Client ID и API key. Она не означает, что система уже загрузила и обработала данные кабинета.



Первичная инициализация — это onboarding-state, а не вся интеграция



Запуск initial sync на этапе 1 нужен как старт интеграционного процесса и как пользовательский сигнал “система начала подготовку”. Полная загрузка всех бизнес-данных относится к следующим этапам.



Основные бизнес-правила этапа

Пользователь не может работать с Ozon без авторизации



Все действия, связанные с seller account и Ozon connection, доступны только авторизованному пользователю.



Ozon credentials должны храниться безопасно



Client ID и API key не должны храниться в открытом виде, не должны возвращаться в ответах API после сохранения и не должны попадать в логи.



Подключение должно иметь статус



Подключение Ozon не должно быть “просто записью в базе”. Оно должно иметь статус, по которому frontend и backend понимают, в каком состоянии находится интеграция.



Initial sync должен иметь статус



Запуск первичной инициализации должен менять состояние системы. Пользователь должен видеть, был ли запуск создан, выполняется ли он, завершён ли он или завершился ошибкой.



Повторный вход не должен сбрасывать состояние



После повторного входа пользователь должен видеть уже созданный seller account, сохранённое подключение и актуальный статус подключения.



Минимальный набор пользовательских состояний



На уровне продукта должны существовать хотя бы следующие смысловые состояния:



Пользователь

не зарегистрирован;

зарегистрирован, но не авторизован;

авторизован.

Подключение Ozon

не создано;

сохранено, но не проверено;

проверяется;

валидно;

невалидно.

Первичная инициализация

не запускалась;

ожидает выполнения;

выполняется;

завершена;

завершена с ошибкой.

Критерии готовности этапа



Этап 1 считается завершённым, если выполняются все условия ниже:



пользователь может зарегистрироваться через UI;

пользователь может войти через UI;

пользователь получает доступ к своей защищённой зоне;

seller account создаётся и привязывается к пользователю;

пользователь может сохранить Ozon credentials;

credentials сохраняются безопасно;

пользователь может проверить подключение;

система корректно показывает результат проверки;

пользователь может запустить initial sync;

система отображает статус initial sync;

после повторного входа пользователь видит своё текущее состояние, а не начинает сценарий заново.

Риски, которых нужно избежать на этапе 1

Подмена этапа 1 полноценной интеграцией



Главный риск — начать тащить в этап 1 всю загрузку данных Ozon и тем самым сильно раздуть объём.



Смешение user и seller account



Нужно сохранить различие между пользователем как субъектом входа в систему и seller account как рабочим объектом бизнеса.



Небезопасное хранение секретов



Недопустимо хранить API key в открытом виде или возвращать его на frontend.



Отсутствие статусной модели



Если не ввести статусы подключения и initial sync, frontend не сможет корректно строить onboarding-flow.



Итоговая формулировка этапа



Этап 1 — это этап, на котором пользователь впервые становится реальным пользователем продукта: он получает доступ в систему, привязывает к ней свой Ozon-кабинет и доводит систему до состояния готовности к загрузке данных.
