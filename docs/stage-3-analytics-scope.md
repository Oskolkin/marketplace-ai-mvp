# Stage 3 - Analytics Scope

## Назначение документа

Этот документ фиксирует аналитический scope этапа 3 для MVP и задает границы реализации, чтобы команда могла двигаться последовательно и без двусмысленностей.  
Документ не описывает код, SQL, API-контракты и не меняет runtime-логику.

## Цель этапа 3

Собрать и отобразить базовую operational-аналитику для селлера на основе уже реализованного ingestion-контура этапа 2, с фокусом на:

- account-level обзор динамики;
- SKU-level вклад в результат;
- видимость текущих остатков и их покрытия.

Результат должен быть пригоден для ежедневного операционного контроля, но не позиционироваться как полноценная BI/финансовая система.

## Почему нужен отдельный аналитический scope

- Этап 2 уже дал ingestion-фундамент (`products`, `orders`, `sales`, `stocks`, `sync_jobs`, `import_jobs`, `sync_cursors`, `raw_payloads`), но без жестких границ этап 3 легко расползается в большой BI-продукт.
- Отдельная фиксация scope защищает roadmap от преждевременного включения сложных слоев (recommendations, AI-chat, продвинутая финансовая аналитика, конкурентные модели).
- На текущем аккаунте разработчика нет реальных данных Ozon, поэтому нужны заранее зафиксированные правила dev/synthetic стратегии, чтобы не блокировать разработку и проверку интерфейсов.

## In scope

Ниже перечислено только то, что входит в этап 3.

### Account-level

- выручка;
- количество заказов;
- возвраты;
- отмены;
- day-to-day динамика;
- week-to-week динамика.

### SKU-level

- выручка по SKU;
- количество заказов по SKU;
- доля SKU в общем результате;
- вклад SKU в изменение результата;
- текущий остаток;
- days of cover.

### Frontend

- dashboard v1;
- KPI cards;
- таблица SKU;
- таблица stocks.

## Out of scope

В шаг 1 этапа 3 и в зафиксированный scope этапа 3 не входят:

- alerts;
- recommendations;
- AI-chat;
- объяснение причин просадки;
- приоритетизация SKU;
- critical SKU screen;
- рекламные сигналы как продуктовый слой;
- сложная маржинальность;
- продвинутые временные ряды;
- конкурентная аналитика;
- полноценная финансовая отчетность / P&L.

## Account-level метрики

На уровне аккаунта в этапе 3 считаются и показываются только operational KPI:

- выручка (агрегированно по периоду);
- количество заказов;
- возвраты;
- отмены;
- изменение ключевых KPI day-to-day;
- изменение ключевых KPI week-to-week.

Требование к трактовке: это operational-слой для мониторинга динамики, не бухгалтерская и не финальная финансовая отчетность.

## SKU-level метрики

На уровне SKU в этапе 3 считаются и показываются:

- выручка по SKU;
- количество заказов по SKU;
- доля SKU в общем результате аккаунта;
- вклад SKU в изменение общего результата (динамика);
- текущий остаток по SKU;
- days of cover как операционная оценка покрытия.

Требование к трактовке: значения используются для оперативного контроля ассортимента и стока, без заявлений о прогнозной точности enterprise-уровня.

## Frontend surfaces

В рамках этапа 3 создаются/дорабатываются только следующие поверхности:

- `dashboard v1` как основной экран operational-обзора;
- `KPI cards` для account-level показателей;
- `таблица SKU` для SKU-level вклада и динамики;
- `таблица stocks` для текущих остатков и покрытия.

Эти поверхности должны опираться на уже реализованный ingestion и не включать out-of-scope функциональность.

## Источники данных и ограничения Ozon

- Этап 3 строится на данных Seller API и уже реализованном ingestion-контуре проекта.
- Продукты, заказы, продажи и остатки являются core data для первой версии аналитики.
- Доступ к данным Ozon в проекте осуществляется через `Client ID` и `API key` в рамках уже реализованного контура подключения.
- Остатки трактуются как operational snapshot, а не как strict realtime truth, поскольку обновления на стороне Ozon могут иметь лаг.
- Рекламный контур вынесен за рамки текущего шага (реклама не включается в продуктовый аналитический слой этапа 3).
- Финансовые KPI в этом этапе ограничены operational-аналитикой и не претендуют на полный расчет прибыли, комиссий и выплат.

## Ограничения текущего проекта

- Текущая реализация согласуется с этапом 2: ingestion-контур есть, но аналитический продуктовый слой только формируется.
- У разработчика сейчас только креды пустого Ozon Seller аккаунта, поэтому live-данные недостаточны для проверки всех KPI и таблиц.
- Scope этапа 3 должен оставаться узким и операционным, чтобы избежать перерастания в "пол-BI-системы" до закрытия базового dashboard v1.

## Dev / synthetic data strategy

- Поскольку реальный Ozon аккаунт пустой, этап 3 нельзя валидировать только на live-данных.
- Для реализации и проверки `dashboard v1` потребуется synthetic/demo data.
- Synthetic data должна генерироваться на уровне source/operational tables или совместимого dev-seed слоя.
- Synthetic data обязательна для `products`, `orders`, `sales`, `stocks`, а также для проверки day-to-day и week-to-week динамики.
- Synthetic data не должна подменять production ingestion-логику; это отдельный dev-инструмент для локальной разработки и тестирования аналитических поверхностей.

## Результат этапа 3

Этап 3 считается завершенным, когда:

- зафиксированные account-level и SKU-level метрики стабильно доступны для dashboard v1;
- реализованы и проверяемы `KPI cards`, `таблица SKU`, `таблица stocks`;
- проверки проходят как на live-данных (где возможно), так и на synthetic/dev data;
- out-of-scope направления не внедряются в рамках этапа 3.

Итог этапа 3: рабочий operational analytics v1, согласованный с текущим ingestion-фундаментом этапа 2 и ограниченный четким scope.
